# TASKS_fast — быстрый ввод: клавиатура, ручная правка, мультистрока, парсер-справочник, справка

> Атомарный план по правкам «Быстрого ввода» (`QuickInputContent` в `src/pages/AppShell.tsx`).
> Каждый пункт — самостоятельная задача: «читать» (контекст, не менять) → «редактировать/создать» (результат).
> Код не пишем, только план. Правки БД (коллекции/поля PocketBase) выполняет пользователь — только указано что/где менять.
> После реализации всех пунктов — выполнить `graphify update .`, затем обновлён `.cursor/rules/stack.mdc` (см. низ файла).

## Решённые вопросы (уточнено с пользователем)

- Материал/тариф — **всегда отдельная последняя строка** ввода (минимум 2 строки: 1+ строка работы + 1 строка материал/тариф). Однострочный ввод «всё в одной строке» (старый формат) **не поддерживаем**.
- Поля внутри строки работы распознаются **по ключевым словам, порядок не важен** (как сейчас в `parseQuickText`), а не по фиксированной позиции через запятую.
- Ручное исправление поля в карточке — **кастомный dropdown** со списком из справочника (как на Главной), не голый текст-инпут.
- Кнопка «Проверить» **повторно прогоняет распознавание** тех же данных карточки через тот же парсер/валидатор, что и «Распознать» (единая функция, не отдельная ветка логики).
- Кнопка «Разобрать» → переименовать в **«Распознать»** везде в этом блоке UI.
- «Шумовая полоса» в `marking_numbers` — без числового `number`, только текст; алиасы (`шум полоса`/`шумка`/`шумполоса`/`шумовая полоса`) маппятся напрямую на это текстовое значение, не на `1.18`.
- Материалы — сравнение алиасов с `materials.name` тоже регистронезависимое (как и остальные поля).
- Данные сверены напрямую с реальной БД (`backup_db/data.db`, локальный SQLite-бэкап PocketBase, добавлен в `.gitignore`). Ключевой вывод сверки: сабтипы `1.18`/`1.24.1`/`1.24.2` — это отдельные записи `marking_numbers` с одинаковым `number` (не `marking_types`); материал в БД называется «Пластик», не «Холодный пластик» (см. «Сверка с реальной БД» в Задаче 4).

## Задача 2. Ручная правка распознанных/нераспознанных полей + переключение кнопки «Добавить в таблицу» ↔ «Проверить»

> Реализуется совместно с Задачей 3 (мультистрока), так как меняется структура состояния карточек. Смотреть обе перед правками.

- **Читать:** `src/pages/AppShell.tsx` — блок рендера `FIELD_LABELS.map(...)` (L155‑176) и `handleAdd()` (L86‑88); `src/pages/HomePage.tsx` — `DropdownCard()`/`DesktopDropdown()` (см. Блок 0) как образец тап-по-полю → открыть список опций справочника → выбор.
- **Дизайн:**
  - Каждое поле карточки (место / № разметки / тип / кол-во / материал / тариф) становится тappable: тап открывает `DropdownCard`/`DesktopDropdown` с опциями из справочника (`dict.locations`, `dict.markingNumIds`+`markingNumMeta`, `typesForMarking(...)`, `dict.materials`) — для «кол-во» и «тариф» (числовые поля без справочника) — компактный текстовый инпут вместо dropdown.
  - Состояние карточки хранит флаг `dirty` (сброшен после успешного распознавания/проверки, взводится при любой ручной правке поля).
  - Кнопка под карточкой:
    - нет карточек / текст не распознавался → «Распознать» (запускает парсер, создаёт карточки, `dirty = false`).
    - карточки есть, `dirty === false` → «Добавить в таблицу» (сразу отправляет карточки в `onAdd`).
    - карточки есть, `dirty === true` (после ручной правки) → «Проверить» (повторно прогоняет карточку через ту же функцию валидации/нормализации, что и при первичном распознавании — см. Задачу 4; по завершении `dirty = false`, кнопка возвращается к «Добавить в таблицу», если карточка валидна).
- **Редактировать:** `src/pages/AppShell.tsx` — `QuickInputContent()` полностью переписывается под карточную модель (см. Задачу 3), логика кнопки/`dirty` — часть этого компонента.

  - `TARIFF_KEYWORD` = `"тариф"`, `TARIFF_UNIT_WORDS`: `рублей`,`руб.`,`руб`,`р.`,`р`.
  - Каждая группа — с комментарием-примером и пояснением, что редактируется вручную (без пересборки логики). При появлении новых записей `marking_numbers`/`marking_types`/`materials` в БД — новые алиасы дописываются сюда же вручную (id/название смотреть в админке PB).

### 4.1–4.6 Логика парсера (использует справочник из 4.7)

- **Читать:** `src/lib/db.ts` (`Dictionaries`, `MarkingNumberItem`, `MarkingTypeItem`, `sortedMarkingNumbers`, `markingTypesByNumberId`, `markingNumberIdByNumber`), `src/lib/sync.ts` (`useDictionaries`, `markingNumHasTypes`), `src/lib/markingValue.ts` (`locationZone`) — для согласованной терминологии «место → зона».
- **Создать:** `src/lib/quickInputParser.ts` — вся логика разбора и валидации, импортирует данные из `quickInputKeywords.ts` и типы/справочники из `db.ts`. Экспортирует:
  - `type ParsedField<T> = { value: T; recognized: boolean }` — обёртка для отображения «распознано / не распознано» на каждое поле карточки (замена текущей эвристики `hasVal` в `AppShell.tsx` L158).
  - `type ParsedQuickRow = { location: ParsedField<string>; markingNum: ParsedField<string>; markingNumberId: ParsedField<string>; markingType: ParsedField<string>; volume: ParsedField<number> }`.
  - `parseWorkLine(line: string, dicts: Dictionaries): ParsedQuickRow` — реализует: 4.1 (место: `LOCATION_KEYWORDS`, регистронезависимо, `\b`-границы слов); 4.2 (номер: сначала точное совпадение с `dicts.markingNumbers[].number`/алиасы `MARKING_NUM_ALIASES` → получаем базовый `number`/список кандидатов `markingNumberId`; если для этого `number` есть `MARKING_NUMBER_VARIANT_ALIASES[number]` — ищем в остатке строки один из вариант-алиасов и **заменяем** кандидата на конкретный `markingNumberId`, иначе — если под этим `number` в БД ровно одна запись, берём её id напрямую, если несколько без явного варианта — поле помечается `recognized: false`, пользователь выбирает вручную через dropdown, см. Задачу 2); 4.3 (типоразмер: если у выбранного `markingNumberId` есть `marking_types` в `dicts.markingTypes`, ищем совпадение по `MARKING_TYPE_SIZE_ALIASES` → сверяем с `marking_types.name`); 4.4 (количество: число + опциональный суффикс из `QUANTITY_UNIT_WORDS`, регистронезависимо, слитно или через пробел, из остатка строки после вычленения места/номера/варианта/типа).
  - `parseMaterialTariffLine(line: string, dicts: Dictionaries): { material: ParsedField<string>; tariff: ParsedField<number> }` — реализует 4.5 (материал по `MATERIAL_KEYWORDS`, сверка с `dicts.materials[].name`) и 4.6 (тариф: `TARIFF_KEYWORD` + число слитно/через пробел, опциональные `TARIFF_UNIT_WORDS`; при отсутствии ключевого слова «тариф» — резервный разбор: последнее число в строке, не совпавшее с количеством/материалом).
  - `parseQuickInput(text: string, dicts: Dictionaries): { workRows: ParsedQuickRow[]; materialTariff: { material: ParsedField<string>; tariff: ParsedField<number> } | null; error?: string }` — верхнеуровневая функция: разбивает на строки (Задача 3), последняя строка → `parseMaterialTariffLine`, остальные → `parseWorkLine`; если строк < 2, возвращает `error`.
  - Эта же `parseQuickInput`/`parseWorkLine`/`parseMaterialTariffLine` используется и кнопкой «Проверить» (Задача 2) — не дублировать логику проверки где-то ещё в `AppShell.tsx`.
- **Редактировать:** `src/pages/AppShell.tsx` — удалить старую `parseQuickText()` (L39‑67) и `FIELD_LABELS` (L71‑77, заменится на рендер по структуре `ParsedQuickRow`/`materialTariff`), переключить `QuickInputContent` на `parseQuickInput` из нового модуля; получить `dicts` через `useDictionaries()` (`src/lib/sync.ts`) внутри `AppShell.tsx`/`QuickInputContent` (сейчас справочник есть только на `HomePage`).
